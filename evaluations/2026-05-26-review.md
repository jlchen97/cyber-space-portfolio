## P0 — Ship blockers

### `[P0]` `NEXT_PUBLIC_SITE_URL` placeholder is the production fallback in 3 files

- **Where:** `app/layout.tsx:26–28`, `app/sitemap.ts:10–12`, `app/robots.ts:3–5`
- **Why it matters:** A deploy without that env var bakes `https://TODO_SITE_URL.com` into:
  - the canonical `<link rel="canonical">` tag (search engines will dedupe to the wrong origin),
  - every `<url>` in `sitemap.xml`,
  - `metadataBase` (so OG image and twitter card URLs resolve against the placeholder — LinkedIn/Slack unfurls break entirely).
- **Fix:** Two layers of defense:
  1. In the bootstrap/deploy scripts (`bootstrap.sh`, `deploy.sh`), refuse to build if `NEXT_PUBLIC_SITE_URL` is unset.
  2. In code, throw at build time rather than fall back silently:
     ```ts
     const SITE_URL = (() => {
       const raw = process.env.NEXT_PUBLIC_SITE_URL?.replace(/\/$/, "");
       if (!raw && process.env.NODE_ENV === "production") {
         throw new Error("NEXT_PUBLIC_SITE_URL must be set for production builds");
       }
       return raw || "http://localhost:3000";
     })();
     ```
     Centralize this in `lib/site.ts` so all three callers import the same value.

## P1 — High-impact, low-effort

### `[P1]` `glowy-waves-hero.tsx` has the same rAF + scroll-wasteful pattern Cobe just fixed

- **Where:** `components/ui/glowy-waves-hero.tsx:63–279`
- **Why it matters:** This canvas is the full-viewport hero on `/projects`. Once the user scrolls past it into the dossier list, the `animate()` rAF keeps running, `ctx.fillRect`-ing across `window.innerWidth × innerHeight` every frame and re-drawing 5 sine waves. There's also a window-level `mousemove` handler (line 205) that fires on every move regardless of whether the canvas is visible. The `prefers-reduced-motion` branch only attenuates the wave influence; it doesn't stop the loop.
- **Fix:** Mirror the Cobe pattern. Wrap the canvas with an `IntersectionObserver`, toggle a `visible` ref, and short-circuit the rAF tail (`if (!visible) { animationId = null; return; }`). Remove the `mousemove` listener when offscreen as well. If `prefers-reduced-motion` is set, render a static one-frame draw and skip the rAF entirely — the visual cost of one frame is the same as no animation but with the calm of no GPU loop.

### `[P1]` `SpecialText` scramble and global `scroll-behavior: smooth` still ignore reduced-motion

- **Where:** `app/globals.css:141–143`, `components/ui/special-text.tsx` (entire file)
- **Why it matters:** Vestibular-sensitive users still get the scrambled landing-name reveal and smooth-scroll on every anchor jump. The "honor reduced-motion" finding from the baseline is half-fixed.
- **Fix:**
  ```css
  @media (prefers-reduced-motion: reduce) {
    html { scroll-behavior: auto; }
    .animate-blink, .animate-spotlight, .animate-pulse { animation: none !important; }
  }
  ```
  In `SpecialText`, short-circuit:
  ```ts
  const reduce = typeof window !== "undefined" &&
    window.matchMedia("(prefers-reduced-motion: reduce)").matches;
  if (reduce) return <span className={className}>{children}</span>;
  ```

### `[P1]` JSON-LD `Person` schema is missing

- **Where:** `app/layout.tsx` (no `<script type="application/ld+json">` anywhere — `grep "application/ld+json"` returns nothing)
- **Why it matters:** A portfolio is a textbook use case for the schema.org `Person` type — `name`, `jobTitle`, `alumniOf`, `worksFor`, `sameAs` (LinkedIn), `image`. Google's Knowledge Panel + rich-result eligibility hinges on having this declared.
- **Fix:** In `app/layout.tsx`, render a script tag near the body:
  ```tsx
  <script
    type="application/ld+json"
    dangerouslySetInnerHTML={{
      __html: JSON.stringify({
        "@context": "https://schema.org",
        "@type": "Person",
        name: "Jian Chen",
        jobTitle: "Mechanical & Robotics Engineer",
        worksFor: { "@type": "Organization", name: "Level Up Live" },
        alumniOf: { "@type": "CollegeOrUniversity", name: "University of Washington" },
        url: SITE_URL,
        image: `${SITE_URL}/opengraph-image`,
        sameAs: ["https://www.linkedin.com/in/jianlchen/"],
      }),
    }}
  />
  ```

### `[P1]` `<nav>` and `<footer>` are duplicated across all 5 page files

- **Where:** `NAV_LINKS` declared 5 times (`app/page.tsx:36`, `app/projects/page.tsx:8`, `app/projects/[slug]/page.tsx:75`, `app/experience/page.tsx:7`, `app/experience/[slug]/page.tsx:8`); the footer JSX is duplicated in all 5 files (`© 2026 JIAN CHEN. SYSTEMS NOMINAL` literal appears 5×).
- **Why it matters:** Any nav or footer copy change requires editing 5 files in lockstep — and the per-page `active` flag is the only meaningful variance. This is exactly what `app/layout.tsx` is for, or at minimum a shared `<SiteNav active={...} />` / `<SiteFooter />` component pair.
- **Fix:** Promote nav + footer into `components/site-chrome.tsx` (or split into two files). Read the current pathname server-side (`headers()` + `next/headers`) or pass an `active` prop from each page; either way 5× → 1× source.

### `[P1]` 22 MB of source JPEGs — Amazon Robotics alone is 12 MB

- **Where:** `public/images/experience/amazon-robotics/` (12 MB across ~30 files, top 10 are 630 KB–1.05 MB each)
- **Why it matters:** Even with `next/image` doing per-request transcoding, the *first* visitor on each variant cold-starts the optimizer against the original 1 MB JPEG. Repo clones, `git pull`, and CDN cold-start each pay the full 22 MB. The 11 small AVIFs that were committed are great — but they're in unrelated folders.
- **Fix:** One-time pass with `sharp` / `squoosh-cli`:
  ```bash
  npx @squoosh/cli --mozjpeg '{"quality":80}' \
    --resize '{"enabled":true,"width":2400}' \
    public/images/experience/amazon-robotics/*.jpg
  ```
  Expected: ~12 MB → ~3 MB with no visible quality loss at the rendered sizes (cards are `50vw`, hero is `100vw` capped at 1440 px).

### `[P1]` Alt text is still placeholder-grade for photographic content

- **Where:** `app/experience/page.tsx:246` (`alt={entry.company}` — just "Tesla"), `app/projects/[slug]/page.tsx:369,385` (`alt={`${section.title} 1`}`, `alt={`${section.title} ${i+2}`}` — e.g. `"Pod Overview 3"`)
- **Why it matters:** Screen-reader output for the project gallery becomes "Pod Overview 3, Pod Overview 4, Pod Overview 5" — useless. Adds nothing for image search either.
- **Fix:** Extend the data layer. Add an `alt?: string` per image entry in `lib/projects.ts` and `lib/experience.ts`, falling back to the current templated string when unset, but populate it for the hero shots that actually describe something (e.g. *"Hyperloop competition pod chassis exiting the test tube at SpaceX, July 2017"*). Decorative-only images on the `[slug]` heroes already correctly use `alt=""` — good.

## Ideas

### `[Idea]` Promote the four pages' "telemetry" copy into a single `lib/copy.ts`

The repeated `"SYSTEMS NOMINAL"`, `"DEPLOY_NN"`, `"ART-NN"`, `"OPEN CHANNEL"` strings are part of the brand voice but live as inline literals in 5 files. A central glossary doesn't just DRY them up — it makes the cyber-vocabulary auditable in one place when you ever want to add `"MISSION ARMED"` or shift the punctuation style.

### `[Idea]` Treat `lib/projects.ts` + `lib/experience.ts` as MDX

Each `section.body` is a single string — no formatting, no inline `<code>`, no links. Converting to MDX (per-project `lib/projects/<slug>.mdx`) would unlock callouts, code samples (a CAD constraint snippet), and inline images, and would shrink the giant TS modules. Next 16's App Router has first-class MDX support.

### `[Idea]` Hot-key search palette (cmd-K)

A portfolio's whole job is letting a hiring manager find one specific story. A `cmd-K` palette indexing `PROJECTS` + `EXPERIENCE` (title, company, tags) would feel native to the "mission control" framing — and you already have `tags` and `tools` arrays primed for it.

### `[Idea]` Resume page that's *the* resume

Instead of the Wix PDF, render a print-stylesheet version of `lib/experience.ts` at `/resume`. `@media print` strips the nav + animations; the cyber-green tokens become black-on-white; `window.print()` produces a clean A4. Eliminates the rot risk and means the resume is always current with the site.
