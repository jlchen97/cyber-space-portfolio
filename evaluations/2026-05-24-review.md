## P0 — Ship blockers / actively hurting

### `[P0]` All site images use raw `<img>`, not `next/image`

- **Where:** `app/projects/[slug]/page.tsx`, `app/projects/page.tsx`, `app/experience/[slug]/page.tsx`, `app/experience/page.tsx`
- **Evidence:** `grep "from 'next/image'"` returned 0 matches; 16+ raw `<img>` tags found across the four pages.
- **Why it matters:** With 22 MB of mostly-JPEG photography (largest single asset ~1 MB), no `next/image` means no automatic resizing, no AVIF/WebP negotiation, no lazy-loading-by-default, and no responsive `srcset`. On a project detail page with a gallery, this is the dominant cause of slow load and CLS.
- **Fix:** Replace `<img>` with `next/image`. For the hero and above-the-fold gallery shots, pass `priority`; for everything below, default lazy loading is correct. Provide `sizes` (e.g. `"(min-width: 1024px) 50vw, 100vw"`) so the browser picks the right candidate. Next's built-in optimizer will produce AVIF + WebP automatically.
- **Estimated impact:** Likely 50–80% reduction in image bytes on first paint of the heaviest pages.

### `[P0]` `next.config.ts` is empty — no image, header, or compression policy

- **Where:** `next.config.ts` (8 lines, no config keys set)
- **Why it matters:** Once `next/image` is adopted, the optimizer needs `images.formats: ['image/avif', 'image/webp']` and possibly `deviceSizes`/`imageSizes` tuned for the gallery widths actually used. No `headers()` block means no security headers (CSP, X-Frame-Options, Referrer-Policy) and no long-cache headers for `/images/*`. No `poweredByHeader: false` so the `X-Powered-By: Next.js` header leaks framework info.
- **Fix:** Add an `images` block, a `headers()` function for `/_next/static` and `/images/*` (1-year immutable), and basic security headers.

## P1 — High-impact, low-effort

### `[P1]` Two Spline scenes load on the landing page; no `dynamic` import, only `React.lazy`

- **Where:** `components/ui/splite.tsx` uses `React.lazy`, but `app/page.tsx` mounts both `SPLINE_SCENE` (robot) and `SPLINE_BRAIN` together.
- **Why it matters:** `React.lazy` defers the JS chunk but does NOT prevent both scenes from initializing in parallel as soon as the home page mounts. Spline's `@splinetool/runtime` is ~hundreds of KB gzipped per scene, and each scene fetches its own `.splinecode` asset. On mobile or slow connections this dominates LCP and consumes battery via WebGL.
- **Fix:** (a) Use `next/dynamic` with `ssr: false` for the brain scene specifically. (b) Wrap the brain in an `IntersectionObserver` so it only initializes when scrolled into view (the brain is well below the fold). (c) Consider a lower-poly placeholder image until the viewport hits the Professional section.

### `[P1]` Cobe globe (`/experience` hero) is heavy WebGL with no viewport gating

- **Where:** `components/ui/cobe-globe.tsx` (client component, runs on mount)
- **Why it matters:** Cobe paints to a canvas with `requestAnimationFrame` continuously. If the user scrolls past it, it keeps spinning and consuming CPU/GPU.
- **Fix:** Pause the rAF loop when the canvas is not intersecting (`IntersectionObserver` → toggle a `paused` ref). Same pattern for `glowy-waves-hero.tsx` on `/projects`.

### `[P1]` `html { scroll-behavior: smooth }` doesn't respect `prefers-reduced-motion`

- **Where:** `app/globals.css` line 142
- **Why it matters:** Vestibular-sensitive users get unwanted smooth scrolling. Also affects the `blink` and `spotlight` keyframes (lines 119–139).
- **Fix:**
  ```css
  @media (prefers-reduced-motion: reduce) {
    html { scroll-behavior: auto; }
    .animate-blink, .animate-spotlight { animation: none !important; }
  }
  ```
  Also short-circuit the `SpecialText` scramble for the same media query.

### `[P1]` SEO surface is minimal — no OpenGraph, Twitter cards, sitemap, or robots

- **Where:** `app/layout.tsx` exports only `title` and `description`.
- **Why it matters:** When someone shares the URL on LinkedIn, Slack, or iMessage, the unfurl will be a generic favicon + plain title. For a portfolio used in hiring conversations, a polished card with a hero image is table stakes.
- **Fix:** Extend `metadata`:
  ```ts
  openGraph: { title, description, url: "https://...", siteName: "Jian Chen", images: [{ url: "/og.png", width: 1200, height: 630 }], type: "website" },
  twitter: { card: "summary_large_image", title, description, images: ["/og.png"] },
  alternates: { canonical: "https://..." },
  metadataBase: new URL("https://...")
  ```
  Add `app/sitemap.ts` (App Router native — emits `sitemap.xml`) listing the experience and project slugs. Add `app/robots.ts` with a sensible default.

### `[P1]` Missing alt text on photographic `<img>`

- **Where:** 16+ `<img>` tags across project/experience pages. From the grep output, `alt` is either short or templated; worth auditing each individually for descriptive value, not just presence.
- **Why it matters:** Screen-reader users and image-search both penalize empty or generic alt. Hero images should describe the subject; decorative spotlights can use `alt=""`.
- **Fix:** Hand-curate `alt` per project section. Where the image is purely decorative (gradient panels, etc.), `alt=""` is correct.

## P2 — Polish & longer-term

### `[P2]` 22 MB of raw JPEGs ship in the repo

- Even with `next/image` doing per-request optimization, the source files themselves are wastefully large for the web. Several Amazon-robotics files are >1 MB at presumably ~1920px wide.
- **Fix:** Run a one-time `sharp` or `squoosh-cli` pass to re-encode source JPEGs at q80 and cap longest edge at 2400px. Keeps Next's optimizer fast and shrinks repo clones.

### `[P2]` `lucide-react@1.14.0` is several major versions behind current

- Brand icons were dropped in v1, hence the inline LinkedIn SVG. Worth checking whether newer lucide-react has reasonable API parity and whether the dropped brand icons are now provided by a sibling package (e.g. `simple-icons`).
- Low priority — current code works.

### `[P2]` Resume URL is a raw 36-char UUID Wix file URL

- `app/page.tsx` line 30: `RESUME_URL = "https://cf9590f3-77e3-40c9-8362-58710c7c207a.filesusr.com/..."`
- This URL will rot the day the Wix site is removed. Consider committing `public/resume.pdf` and serving it from your own origin (`/resume.pdf`), with the Wix URL as a fallback.

### `[P2]` `glass-panel` uses `backdrop-filter: blur(20px)` site-wide

- Heavy on lower-end mobile GPUs, especially on the fixed nav which composites every scroll frame. Consider `blur(12px)` or feature-detecting `@supports (backdrop-filter)` and falling back to a flat translucent color.

### `[P2]` No `loading.tsx` or `error.tsx` at route level

- Per-route loading skeletons would smooth navigation between `/projects/[slug]` pages, especially while Spline/Cobe JS is hydrating elsewhere. Error boundaries protect against a single broken project entry taking down the route.

### `[P2]` `JOURNEY_FEATURED = ["Level Up Live", "Amazon Robotics", "Tesla"]` uses non-null assertion on `find`

- `app/page.tsx` line 44: `EXPERIENCE.find((e) => e.company === name)!`
- A typo in one of the strings silently becomes a runtime crash with no error message. Replace `!` with a runtime check or a typed lookup table.

### `[P2]` Smooth-scroll anchor offset is hardcoded to 100px

- `.anchor-offset { scroll-margin-top: 100px }` in `globals.css`. If the nav height ever changes (e.g. for a campaign banner), every anchor breaks. Worth exposing as a CSS variable: `scroll-margin-top: var(--nav-height, 100px)`.

## Ideas

### `[Idea]` Add `view-transitions` between project detail pages

- Next 16's experimental view-transitions API would give the hero image a smooth morph from card → detail page. Fits the "aerospace control panel" aesthetic well.

### `[Idea]` Telemetry as a feature

- The site's whole vibe is "mission control." A real, opt-in `web-vitals` widget in the corner showing LCP / CLS / FID for the current page would be on-brand AND useful for the bi-daily perf review.

### `[Idea]` Replace the Cobe globe markers with a live "currently in X" pin

- Wire it to a single static `lib/status.ts` const for now; over time, a tiny edge function could update it. Sells the "active operator" framing.

### `[Idea]` Print stylesheet for the resume page

- If you ever route resume PDFs through the site, a `@media print` block (no nav, no animations, ink-friendly contrast) is a nice touch.

## Suggested order of attack

If you only do three things this round:

1. Migrate `<img>` → `next/image` across the four pages (P0)
2. Set up `metadata.openGraph` + `app/sitemap.ts` + `app/robots.ts` (P1)
3. Gate the Cobe globe and brain Spline behind `IntersectionObserver` (P1)

Together these probably move Lighthouse Performance from "meh" to "good" and Social/SEO from "broken share previews" to "polished."
